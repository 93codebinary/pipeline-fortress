## Code Contributions

The ten minimum requirements for contributing to this project are:

1. All Pull Requests **must** be based off the `master` branch.
2. Pipeline YAML, Python scripts, and Kubernetes manifests **must** follow consistent formatting and naming conventions used throughout the project.
3. All PRs **should** have a dedicated scope — do not mix unrelated changes (e.g. do not combine a new security gate with a Dockerfile fix in the same PR).
4. All new and changed pipeline stages **should** have a corresponding test or validation step that proves the gate works correctly.
5. New security gates **must** include a severity threshold, an enforcement step with `exit 1`, and a report upload with `if: always()`.
6. Noise generated by AI tools (unnecessary comments, hallucinated CVE IDs, unverified fix suggestions) **must** be reviewed and removed before opening a PR.
7. All pipeline status checks **must** pass for the last commit within your PR — all 8 security gates (Gitleaks, Semgrep, Trivy FS, Docker Build, Trivy Image, Checkov, SBOM, ZAP) must be green.
8. All Git commits within a PR **must** be [signed off](https://git-scm.com/docs/git-commit#Documentation/git-commit.txt--s) to indicate agreement with the [Developer Certificate of Origin](https://developercertificate.org/):
   ```bash
   git commit -s -m "feat: add new security gate"
   ```
9. LLM-generated remediation PRs **must** be labelled `llm-generated` and `requires-human-review` and **must not** be auto-merged — a human must review and approve every AI-generated change.
10. Particularly low-effort contributions (incomplete fixes, trivial text changes, formatting-only changes, or contribution farming) **must not** be submitted as PRs. Security-related contributions must demonstrate genuine understanding of the vulnerability or control being added.

## Pipeline Contribution Guidelines

When contributing a new pipeline stage or modifying an existing one, follow this checklist:

```
☐ Job has a clear name with emoji label (e.g. "🔑 Secrets Scan — Gitleaks")
☐ Job has needs: dependency defined correctly
☐ Scanner runs with exit-code: '0' (never auto-fail — gate manually)
☐ Custom severity gate uses python3 to parse JSON and exit 1 on breach
☐ Report uploaded as artifact with if: always() and retention-days set
☐ New secrets added to GitHub Secrets table in README
☐ Version fix applied in any job that reads version.txt
☐ All inline comments updated to explain the change
```

## Security Research Contributions

If you discover a real vulnerability in the pipeline infrastructure (not in Juice Shop — those are intentional):

- **Do not** open a public GitHub issue
- **Do** contact the maintainer via the repository page
- All offensive activity must be confined to the lab sandbox — no real-world targets

---

## GenAI / LLM Contributions

If your contribution was assisted by an LLM (Claude, GPT, Gemini, etc.):

- Disclose which model was used and what it generated in your PR description
- Verify all CVE IDs referenced actually exist — no hallucinated identifiers
- Remove any source code snippets the LLM may have reproduced from external sources
- Strip all secrets and credentials from any LLM-generated output before committing
- All LLM-generated fixes must be reviewed by a human security engineer before merge

## Getting Started

```bash
# Fork and clone the repo
git clone https://github.com/93codebinary/pipeline-fortress.git
cd pipeline-fortress

# Create a feature branch
git checkout -b feat/your-contribution

# Make your changes
# ...

# Sign off your commit
git commit -s -m "feat: description of your change"

# Push and open a PR against master
git push origin feat/your-contribution
```
